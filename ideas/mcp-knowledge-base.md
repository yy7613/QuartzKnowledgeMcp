# アイデアメモ

## やりたいこと（粒度不問）
- MCP サーバーに関する情報を整理して参照できるナレッジベースを作る
- サーバーごとの用途、提供ツール、導入手順、認証方式、対応クライアント、利用例をまとめる
- 「どの MCP サーバーを選べばよいか」「どう設定すればよいか」を素早く判断できるようにする
- 単なるリンク集ではなく、比較しやすい形式で情報を蓄積したい
- 将来的には検索やカテゴリ分類で必要なサーバーを探しやすくしたい
- 可能であれば、各 MCP サーバーのサンプル設定や接続時の注意点も残したい

## まずこれだけ動けば成功と思う最小の状態
- MCP サーバーを複数件登録できる
- 各サーバーについて最低限のメタデータを保存できる
  - 名前
  - 概要
  - タグ（1 件以上必須、最大 5 件、後から変更可能）
  - 提供ツールの一覧
  - セットアップ手順
  - 認証の有無
  - 参考リンク
  - 更新履歴
- キーワード検索またはカテゴリ絞り込みで目的のサーバーを見つけられる
- 詳細ページまたは詳細表示で、導入時に必要な情報を一通り確認できる

## やらなくてよいこと（スコープ外）
- 初期段階での自動クローリングや大規模な収集基盤の構築
- MCP サーバーそのもののホスティングや運用監視
- ユーザーごとの認可や複雑な権限管理
- SNS 的なコメント機能や評価機能
- 初期段階での多言語対応
- 初期段階での AI 要約や自動分類の高度化

## 参考になりそうなもの
- Model Context Protocol の公式仕様
- Microsoft Agent Framework の公式ドキュメントとサンプル
  - 特に .NET sample `ModelContextProtocol`
- EF Core の provider abstraction と SQLite provider のドキュメント
- 公開されている MCP サーバーの GitHub リポジトリ
- 各サーバーの README、セットアップガイド、利用例
- MCP 対応クライアントの設定例
- 既存のツールカタログや技術ナレッジベースの UI パターン

## 現時点の技術方針
- アプリケーションフレームワークは Microsoft Agent Framework を採用する
- 永続化層は EF Core を採用し、DB provider を差し替え可能にする
- デフォルトの開発用 DB は SQLite とする
- Embedding はオプション機能として扱い、MVP の必須要件にはしない
- グラフ構造の表現は必要だが、MVP では専用グラフ DB を前提にしない
- まずは relational DB 上でノードとエッジを保持し、必要なグラフ処理はアプリケーション層で扱う方針を優先する
- ソースコードは GitHub で公開する前提で進める
- ライセンスは MIT License を採用する
- 公開前提のため、認証情報、社内限定情報、ライセンス不明な成果物をリポジトリに含めない

## グラフ構造に関する候補
- 第一候補は QuikGraph
  - .NET 向けの汎用グラフ構造とアルゴリズムを提供する
  - 有向グラフ、無向グラフ、最短経路、探索系アルゴリズムなどを扱いやすい
  - EF Core で保存したノード・エッジを読み出して、アプリケーション層でグラフとして組み立てる構成と相性がよい
- 可視化が必要なら MSAGL を追加候補にする
  - グラフの描画やレイアウト向けで、分析ロジックの中心というより表示用途に向く
- RDF や知識グラフを本格的に扱うなら dotNetRDF を別候補にする
  - RDF、SPARQL、推論、SHACL などの文脈が必要なら有力
  - 一般的なアプリ内のノード・エッジ管理にはやや重い可能性がある

## 現時点の推奨方針
- MVP は EF Core + SQLite を基盤にする
- ノードとエッジの永続化モデルは EF Core で実装する
- グラフ探索や経路計算などが必要な箇所だけ QuikGraph を利用する
- Embedding は後から追加可能なようにインターフェースで分離する
- RDF や専用グラフ DB は、検索要件や推論要件が明確になった段階で再評価する

## 気になる点・未検討事項
- 誰向けのナレッジベースか
  - MCP を初めて触る人向けか
  - 実運用する開発者向けか
- 収集対象をどこまで含めるか
  - OSS の公開サーバーのみか
  - 社内サーバーも含むか
- 情報更新をどう回すか
  - 手入力中心か
  - GitHub などから半自動で取り込むか
- データモデルをどこまで細かく持つか
  - ツール単位まで管理するか
  - サーバー単位のみ管理するか
- タグの扱いをどうするか
  - 自由入力タグのみか
  - 管理されたタグ候補を持つか
- 更新履歴をどこまで保持するか
  - 更新日時と更新者のみか
  - 変更内容の要約まで持つか
  - 差分そのものを持つか
- グラフを何のために使うか
  - 関連サーバーの推薦
  - ツール依存関係の表現
  - クライアント、認証方式、利用用途の関連表現
- MVP のインターフェースをどうするか
  - Web UI
  - CLI
  - API のみ
- 検索要件をどこまで求めるか
  - 単純な全文検索で十分か
  - タグ、カテゴリ、認証方式、用途での絞り込みが必要か
- ナレッジの信頼性をどう担保するか
  - 出典リンクを必須にするか
  - 最終確認日を持つか
- 成功指標を何にするか
  - 必要な MCP サーバーを探す時間の短縮
  - セットアップ時の手戻り削減
  - 調査メモの散逸防止
- GitHub 公開時の運用をどうするか
  - Issue と Pull Request を受け付けるか
  - サンプルデータやスクリーンショットをどこまで同梱するか
  - README や公開用ドキュメントに何を最低限含めるか
