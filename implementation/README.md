# 実装ロードマップ

## 目的
このフォルダは、仕様書をコードへ落とすための段階的な実装計画を管理する。
各フェーズは、最小構成で実際に動くものを積み上げることを最優先とする。

このフォルダは実装の判断履歴と検証記録も兼ねる。現在の公開仕様、運用前提、最新の品質基準を確認するときは、`docs/`、ルート README、`implementation/phase-status.md` を優先すること。
`phase-xx-report.md` に含まれるテスト件数やカバレッジ値は、各フェーズ完了時点のスナップショットであり、常に最新値を表すものではない。

## 実行原則
- 各フェーズの終了時点でアプリケーションはビルド可能であること
- 各フェーズの終了時点で対象機能が最低限動作すること
- 新規ロジックにはユニットテストを必須とすること
- 初期フェーズは具体実装を優先し、後半フェーズで抽象化と差し替え性を追加すること
- LLM と埋め込みはオプション機能とし、未設定でも主機能が成立すること

## MCP 実装参照ルール
- MCP の実装方法は Microsoft Agent Framework .NET sample の `ModelContextProtocol` を第一参照にする
- 基本実装は `Agent_MCP_Server`、認可付き実装は `Agent_MCP_Server_Auth`、Hosted MCP は `ResponseAgent_Hosted_MCP` の構成を参考にする
- 本リポジトリではサンプルの Azure OpenAI 前提をそのまま固定せず、MCP サーバー / ツール構成と認証フローの実装パターンを参照する
- 実装フェーズで MCP ホストやツール公開面を追加する場合は、参照したサンプル名を完了レポートに記録する

## フェーズ進行ルール
- 次のフェーズへ進む前に、直前フェーズのテストカバレッジ率を計測し、目標値と実測値を完了レポートに記録すること
- 次のフェーズへ進む前に、直前フェーズで影響を受ける主要ユースケースの回帰テストを実施し、結果を完了レポートに記録すること
- 次のフェーズへ進む前に、直前フェーズの完了レポートを `phase-xx-report.md` として作成すること
- 次のフェーズへ進む可否は `phase-status.md` で管理し、カバレッジ、回帰テスト、完了レポートの 3 条件が揃うまで次フェーズを開始しないこと

## カバレッジ方針
- 各フェーズの最低ラインは `phase-status.md` の `Coverage target` を基準とする
- Phase 13 以降は最終基準線として行カバレッジ 85% 以上を維持し、低下した場合は原因と許容理由を完了レポートへ記録する
- カバレッジは必ず `dotnet test --collect:"XPlat Code Coverage"` の最新レポートを使い、行 / 分岐の両方を完了レポートへ記録する

## ドキュメント差異確認手順
- 完了レポートでは、参照したドキュメント / 仕様と実装済み挙動を突き合わせ、差分があれば `ドキュメントとの差異` へ明記する
- ランタイム スモークまたは Sample で実際に確認した範囲を完了レポートへ記録する
- ドキュメントに未記載の暫定判断や SDK 公開面の差異がある場合は、次フェーズへの申し送り事項にも残す

## 共通完了条件
- `dotnet build` が成功する
- `dotnet test` が成功する
- 追加したユースケースに対するユニットテストが存在する
- テストカバレッジ率の目標値と実測値が記録されている
- 主要ユースケースの回帰テスト結果が記録されている
- 当該フェーズの完了レポートが作成されている
- ドキュメントとの差異があればフェーズ完了時に記録する

## 状態管理ファイル
- フェーズ進行の状態管理は [implementation/phase-status.md](implementation/phase-status.md) で行う
- 各フェーズの開始前、完了時、次フェーズ開始判定時に更新する
- `Coverage target` は事前に設定した最小基準とし、変更時は根拠を Notes に残す
- `Next phase unlocked` が `yes` になるまでは次フェーズへ進まない

## 補足
フェーズをまたぐ大規模抽象化は避け、必ず直前フェーズで動いている機能を保ったまま次へ進める。
