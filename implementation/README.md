# Implementation Roadmap

## 目的
このフォルダは、仕様書をコードへ落とすための段階的な実装計画を管理する。
各フェーズは、最小構成で実際に動くものを積み上げることを最優先とする。

## 実行原則
- 各フェーズの終了時点でアプリケーションはビルド可能であること
- 各フェーズの終了時点で対象機能が最低限動作すること
- 新規ロジックにはユニットテストを必須とすること
- 初期フェーズは具体実装を優先し、後半フェーズで抽象化と差し替え性を追加すること
- LLM と Embedding はオプション機能とし、未設定でも主機能が成立すること

## フェーズ進行ルール
- 次のフェーズへ進む前に、直前フェーズのテストカバレッジ率を計測し、目標値と実測値を完了レポートに記録すること
- 次のフェーズへ進む前に、直前フェーズで影響を受ける主要ユースケースの回帰テストを実施し、結果を完了レポートに記録すること
- 次のフェーズへ進む前に、直前フェーズの完了レポートを `phase-xx-report.md` として作成すること
- 次のフェーズへ進む可否は `phase-status.md` で管理し、カバレッジ、回帰テスト、完了レポートの 3 条件が揃うまで次フェーズを開始しないこと

## フェーズ一覧
| フェーズ | テーマ | そのフェーズ終了時点で動くもの |
|:--|:--|:--|
| Phase 01 | ホスト起動と検証基盤 | Web API が起動し、ヘルスチェックとテスト基盤が動く |
| Phase 02 | Bronze 取り込み最小実装 | Bronze の登録と取得が SQLite で動く |
| Phase 03 | Silver 正規化最小実装 | Bronze から Silver 下書きを規則ベースで生成できる |
| Phase 04 | Gold 公開と履歴 | Silver から Gold を公開し、履歴を記録できる |
| Phase 05 | カタログ参照と基本検索 | Gold 一覧、詳細、キーワード検索、タグベースカテゴリ検索、基本フィルタが動く |
| Phase 06 | Gold 更新と履歴参照 | Gold 本体更新、タグ更新、履歴取得が動き、タグ制約が守られる |
| Phase 07 | レイヤ分離の導入 | 振る舞いを保ったまま Domain / Application / Infrastructure に分離される |
| Phase 08 | DB 差し替え境界の導入 | Repository Port と SQLite Adapter を切り替え可能にできる |
| Phase 09 | LLM 抽象と能力照会 | Rule-based 整理、preview、capabilities API、fallback が動く |
| Phase 10 | MAF / Foundry Adapter 導入 | LLM 設定時のみエージェント整理が有効になる |
| Phase 11 | Embedding 抽象と No-op 実装 | Embedding 無効でも動作し、有効化点だけ差し込める |
| Phase 12 | 検索拡張と関係投影 | 高度検索、補完、ファセット、related API が動く |
| Phase 13 | 回帰固定と完成度向上 | 主要フローが回帰テスト込みで安定する |

## 共通完了条件
- `dotnet build` が成功する
- `dotnet test` が成功する
- 追加したユースケースに対するユニットテストが存在する
- テストカバレッジ率の目標値と実測値が記録されている
- 主要ユースケースの回帰テスト結果が記録されている
- 当該フェーズの完了レポートが作成されている
- docs との差異があればフェーズ完了時に記録する

## 状態管理ファイル
- フェーズ進行の状態管理は [implementation/phase-status.md](implementation/phase-status.md) で行う
- 各フェーズの開始前、完了時、次フェーズ開始判定時に更新する
- `Coverage target` は事前に設定した最小基準とし、変更時は根拠を Notes に残す
- `Next phase unlocked` が `yes` になるまでは次フェーズへ進まない

## 補足
フェーズをまたぐ大規模抽象化は避け、必ず直前フェーズで動いている機能を保ったまま次へ進める。
