# ハーネスのルール

## フォルダの使い方
- アイデアは `ideas` にのみ書く
- 仕様書・設計ドキュメントは `docs` にのみ書く
- 技術・設計上の意思決定は `docs/adr` に ADR として記録する
- 実装計画と報告書は `implementation` にのみ書く
- ソースコードとテストは `src` にのみ書く
- docs と実装の差異は `work/gaps.md` に、課題は `work/issues.md` に記録する
- 指示されたフェーズに関係するフォルダ以外のファイルを変更してはならない（`work` への追記は除く）

## フェーズの進め方
- 1 フェーズずつ進める。複数フェーズを同時に実装してはならない
- フェーズを開始する前に `implementation/phase-XX.md` の計画を確認すること
- フェーズ完了時は `implementation/phase-XX-report.md` に完了報告書を出力すること
- フェーズ完了時、docs との差異があれば `work/gaps.md` を更新し、持ち越す課題があれば `work/issues.md` に追記すること
- 突破基準を満たせない場合は実装を中断し、理由を報告書に記載してユーザーに判断を仰ぐこと

## 禁止事項
- 既存のテストを削除・スキップしてはならない
- 指示されていない機能を追加してはならない
- ハードコードされた認証情報を書いてはならない
- `rules.md` と `readme.md` をエージェントが書き換えてはならない

## 報告のタイミング
- フェーズ完了時
- 突破基準を満たせないと判断した時点
- 仕様書に記載のない判断が必要になった時点
